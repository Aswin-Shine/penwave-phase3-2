Apply order for raw manifests matters — Kubernetes doesn't build a dependency graph the way Terraform does. `kubectl apply -f` on a Deployment that references a ConfigMap that doesn't exist yet doesn't fail; it creates the Deployment with pods stuck in `CreateContainerConfigError` until the ConfigMap shows up. Apply in this order:

```bash

kubectl apply -f k8s/manifests/configmap/

kubectl apply -f k8s/manifests/frontend/
kubectl apply -f k8s/manifests/ingress/
```

## 1. The IRSA wire-up, concretely

`backend/serviceaccount.yaml` has one annotation that does all the work:

```yaml
annotations:
  eks.amazonaws.com/role-arn: "arn:aws:iam::<ACCOUNT_ID>:role/penwave-irsa-backend-prod"
```

Replace `<ACCOUNT_ID>` with the real value from `terraform output backend_irsa_role_arn`. This annotation is read by EKS's Pod Identity webhook at pod creation time — it's the only place the connection between "this ServiceAccount" and "this IAM role" exists in the Kubernetes layer. Everything else (the trust policy's `sub` claim condition) lives in Terraform, on the AWS side. Both halves have to agree: this ServiceAccount's `name: penwave-backend` and `namespace: penwave` must exactly match `backend_sa_name` and `backend_namespace` in `terraform/modules/irsa/variables.tf`. A mismatch on either side produces an `AccessDenied` on `AssumeRoleWithWebIdentity` with no further detail — see `docs/irsa-deep-dive.md` from the Terraform delivery for the full failure-mode breakdown.

`automountServiceAccountToken: true` in the Deployment's pod spec is required — without it, no token gets mounted and the pod has no way to call STS at all, regardless of the annotation.

## 2. Why secrets are split across two different mechanisms

You'll notice `penwave-backend-secrets` (from `ExternalSecret`) and `penwave-backend-connection-strings` (manually applied) are two separate Kubernetes Secrets, both referenced via `envFrom` in the Deployment.

**`penwave-backend-secrets`** holds atomic values that exist as-is in AWS Secrets Manager: `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, `COOKIE_SECRET`, `METRICS_SECRET`, `DB_PASSWORD`, `REDIS_AUTH_TOKEN`. External Secrets Operator pulls these on a 1-hour refresh cycle. If you rotate a value in Secrets Manager, it propagates into the cluster automatically — though the running pods still need a restart to actually pick up the new env var, since env vars are injected once at container start, not live-reloaded.

**`penwave-backend-connection-strings`** holds `DATABASE_URL` and `REDIS_URL` — full connection strings, not atomic secrets. These are *assembled* from a password (Secrets Manager) plus a hostname (a Terraform RDS/Redis module *output*, not a secret). ESO's `ExternalSecret` resource can technically template multiple remote values together, but doing so here means hardcoding the RDS/Redis endpoint string directly into the `ExternalSecret` manifest — which would need to be regenerated by hand every time Terraform recreates the database (e.g. a fresh `terraform apply` for a new 2-day session gets a new RDS endpoint). That's strictly worse than just creating one plain Secret with a short script that reads live Terraform outputs:

```bash
DB_PASSWORD=$(aws secretsmanager get-secret-value \
  --secret-id penwave/prod/app-secrets \
  --query SecretString --output text | jq -r .DB_PASSWORD)

REDIS_AUTH_TOKEN=$(aws secretsmanager get-secret-value \
  --secret-id penwave/prod/app-secrets \
  --query SecretString --output text | jq -r .REDIS_AUTH_TOKEN)

RDS_ENDPOINT=$(cd terraform && terraform output -raw rds_endpoint)
REDIS_ENDPOINT=$(cd terraform && terraform output -raw redis_endpoint)

kubectl create secret generic penwave-backend-connection-strings \
  -n penwave \
  --from-literal=DATABASE_URL="postgresql://penwave:${DB_PASSWORD}@${RDS_ENDPOINT}:5432/penwave" \
  --from-literal=REDIS_URL="rediss://default:${REDIS_AUTH_TOKEN}@${REDIS_ENDPOINT}:6379" \
  --dry-run=client -o yaml | kubectl apply -f -
```

This is the one secret in the whole setup that ESO doesn't manage, and that's a deliberate, documented exception rather than an inconsistency you'd have to explain away in an interview.

## 3. Liveness vs readiness — why two different paths, and why it matters

`backend/src/middleware/health.ts` already split this correctly before any Kubernetes work started: `/health` (liveness, no external calls) and `/ready` (readiness, checks Postgres + Redis). The Deployment's probes map directly:

```yaml
livenessProbe:
  httpGet: { path: /health, port: 4000 }
readinessProbe:
  httpGet: { path: /ready, port: 4000 }
```

The reason this split exists: if `livenessProbe` checked the database and RDS had a 10-second network blip, Kubernetes would interpret that as "this container is broken" and kill + restart the pod. That's the wrong response to a transient upstream issue — restarting a process that was never actually broken doesn't fix RDS being briefly unreachable, and a restart storm across all replicas (if RDS hiccups for everyone simultaneously) turns one slow moment into a full outage. `readinessProbe` checking the database is correct, because its failure mode is "stop sending traffic to this pod" — not "kill it" — which is exactly right when the pod itself is fine but a dependency isn't.

The frontend uses `/` for both probes since no dedicated health route exists in the Next.js app (verified — no file under `frontend/src/app` implements one). This isn't a gap I silently patched around; it's the same fallback the Dockerfile's own `HEALTHCHECK` directive already uses.

## 4. The NEXT_PUBLIC_API_URL problem — still unresolved, now blocking

This was flagged as a known architectural gap back in the Phase 1/2 summary and remains unfixed in the application code. It matters more on Kubernetes than it did on EC2, for a specific reason: on EC2 you built one image for one known domain and it stayed that way indefinitely. On Kubernetes, the appeal of the platform is supposed to be that the same image can run anywhere — different environments, PR previews, blue/green domains. `NEXT_PUBLIC_API_URL` being baked into the client JS bundle at Docker build time defeats that entirely, because Next.js inlines `NEXT_PUBLIC_*` vars into browser-side code at build time, not at container start. Setting it as a pod env var in `frontend/deployment.yaml` (which I did, for SSR code paths) does nothing for the already-built client bundle the browser actually executes.

I did not patch `frontend/src/lib/api-client.ts` to use a relative `/api` path, because that's an application code change you didn't ask me to make in this delivery, not a Kubernetes manifest change. The correct fix is still: switch to a relative path, rebuild the image once, and this entire problem disappears — no env var threading needed anywhere in the K8s layer. Until that happens, every image build must pass the correct `--build-arg NEXT_PUBLIC_API_URL=https://penwave.ddns.net/api`, and any change to the domain means a full rebuild + repush, not just a `kubectl apply`.

## 5. Resource sizing — explicitly not measured

You asked for conservative defaults sized for t3.medium (2 vCPU / 4GB). Here's the actual math, so "conservative" isn't just a word:

A t3.medium typically reserves roughly 700–900Mi of memory and 200–300m CPU for system pods before any application workload is scheduled — `kube-proxy`, CoreDNS, the AWS Load Balancer Controller webhook, the EBS CSI node driver, and Auto Mode's own node-management agents. That leaves roughly 3.1–3.3GB and 1.7–1.8 vCPU of genuinely schedulable capacity per node.

Backend: 2 replicas × 256Mi request = 512Mi. Frontend: 2 replicas × 256Mi request = 512Mi. Total requested: ~1Gi across 4 pods on a single node, well within budget — but limits (512Mi × 2 + 512Mi × 2 = 2Gi) leave less headroom if both services burst simultaneously. If you see `Pending` pods with `Insufficient memory` in `kubectl describe pod`, that's Auto Mode correctly deciding to provision a second node, not a misconfiguration — Auto Mode's entire value proposition is doing that scale-out automatically without you defining a node group's max size in advance.

These numbers are not from profiling. If you want to right-size for real, run a load test against the deployed cluster and read actual usage via `kubectl top pods` over a sustained period, then adjust `resources.requests` to roughly the p50 usage and `resources.limits` to roughly p99 — that's the standard methodology, and it's a good thing to actually do during your 2-day window if you want a number you can defend rather than one you copied from a doc.

## 6. NetworkPolicy — what it actually restricts, and what it doesn't

The default-deny policy (`netpol-default-deny.yaml`) blocks all ingress and egress for every pod in the `penwave` namespace unless an explicit allow rule exists. The allow rules then carve out exactly: frontend can receive traffic from anywhere (the ALB targets pods directly via `target-type: ip`, which exists outside the cluster's own pod-identity model, so this rule is intentionally permissive — true restriction at that boundary happens via the security group Terraform already created, not NetworkPolicy); backend can only receive traffic from frontend pods or from the Ingress's direct `/api/*` routing; both pods can reach DNS (port 53, required for any outbound name resolution including pulling images by tag, calling AWS APIs, etc.); backend can reach ports 5432/6379/443 outbound (RDS, Redis, AWS APIs) with no further CIDR restriction inside the policy because that's already handled by the security groups in `modules/security`.

One thing this does **not** do, that the original Phase 3 spec implied it might: NetworkPolicy enforcement on EKS Auto Mode depends on the VPC CNI's network policy support being active. Auto Mode enables this by default as of when this was written [Likely — this is a relatively new default and worth confirming with `kubectl get pods -n kube-system | grep aws-node` and checking the CNI's policy enforcement mode after your cluster is actually up, rather than trusting this blindly]. If policies appear to have no effect (traffic flows that should be blocked aren't), that's the first thing to check, not a sign the YAML is wrong.

## 7. Helm — what's parameterized and why

Image repository/tag, replica counts, resource requests/limits, probe paths and timings, autoscaling thresholds, the IRSA role ARN, and the ACM certificate ARN are all `values.yaml` entries — these are the things that legitimately change between a fresh `terraform apply` (new IRSA role ARN, new RDS/Redis endpoints) or a new image build (new tag).

What's **not** parameterized: the manually-applied connection-strings Secret (intentionally outside Helm's lifecycle, per the reasoning in section 2), and the NetworkPolicy's actual port numbers and pod selector labels (these encode real architectural facts about how Penwave's services talk to each other — backend always listens on 4000, frontend always on 3000 — and templating facts that don't change per-environment just adds indirection without benefit).

Installing:

```bash
helm install penwave-backend ./helm/penwave-backend -n penwave \
  --set image.tag=<your-image-tag> \
  --set serviceAccount.irsaRoleArn=$(cd terraform && terraform output -raw backend_irsa_role_arn)

helm install penwave-frontend ./helm/penwave-frontend -n penwave \
  --set image.tag=<your-image-tag>

helm install penwave-ingress ./helm/penwave-ingress -n penwave \
  --set certificateArn=<your-acm-cert-arn>
```
